## 빠른 도메인 용어 풀이

이 서비스가 다루는 분야는 "미국 항공 자격시험 준비"입니다.

| 항공 용어 | 쉬운 번역 | 엔지니어링 관점에서 왜 중요한가 |
|---|---|---|
| **FAA** | 미국 항공 규제기관 | 교재·시험의 출처를 정의하는 "정답 권위(authority)" |
| **PHAK / AFH / IFH / AIM** | FAA 공식 교재 (PDF) | RAG의 source corpus. 환각 대신 근거로 쓰는 ground truth |
| **PPL / IFR / CPL** | 조종 자격 등급 (초급 → 상급) | 콘텐츠를 가르는 라벨 (난이도·범위) |
| **ACS** (Airman Certification Standards) | 시험 출제 표준 코드 체계 — 영역 → 태스크 → 요소의 3단 계층 | 문제를 이 표준 코드에 맞춰 만들면, 시험 범위를 빠짐없이 다루는지 확인할 수 있고 엉뚱한 내용이 섞이는 걸 막을 수 있다 |
| **구술시험 (Oral exam)** | 시험관이 말로 묻는 평가 | 학습 자료가 흩어져 있어 직접 다룬 영역 |

## 왜 이런 설계인가 (AI 엔지니어링 관점)

**왜 ACS 코드 기반으로 출제하나?**
ACS는 "무엇을 알아야 하는가"를 계층 코드로 못박은 공식 표준이다. LLM 자유 생성 대신 이 코드에 매핑해서 출제하면 세 가지가 따라온다 — (1) 문제가 실제 시험 범위에 정렬되고, (2) 영역별 커버리지를 수치로 측정할 수 있고, (3) 모델이 엉뚱한 주제로 새는 걸 막는다. 실제 시험에서도 시험관(DPE, 지정 시험관)이 ACS 코드를 기준으로 문제를 내기 때문에, 같은 코드에 맞춰 출제하면 실제 시험과 같은 틀을 따르게 된다.

**왜 FAA Reference(출처 조회) 기능이 따로 있나?**
항공은 전문용어와 법규(14 CFR) 밀도가 높다. 학생에게 필요한 건 챗봇의 의역이 아니라 "정확한 원문 + 페이지 인용"이다. 그래서 문제 생성과 **같은 RAG 파이프라인을 사전(dictionary)처럼 재사용**한다 — 입력 용어를 임베딩해 가장 가까운 공식 청크를 인용 형태로 반환. 유사도(similarity)가 임계값 미만이면 빈 결과를 돌려준다(그럴듯한 오답을 만들지 않음 = 환각 방지).

**왜 문제를 매번 생성하지 않고 "풀(pool)"을 자산으로 두나?**
매 요청 LLM 생성은 비용·지연(latency)·일관성이 모두 약하다. 오프라인 배치로 사전 생성·검수한 문제 풀을 두고, 사용자 피드백(좋아요/싫어요)으로 자동 정제한다(좋은 문제는 풀 편입, 나쁜 문제는 비활성화).

## 개발 방식

코드는 AI LLM으로 작성, 내 역할은 다음과 같았다.

- **무엇을 만들지 정하기** — ACS 정렬 출제, FAA Reference, 풀 자산화 같은 방향 결정
- **아키텍처·트레이드오프 판단** — 풀 vs 매 요청 생성, SPA vs Next.js, 3072차원 임베딩의 인덱스 포기 등
- **위험 우선순위 설정** — 출시 전 LLM 비용 abuse를 1순위 위협으로 잡고 보안 정비
- **문제 진단과 검증** — PKCE race condition 원인 추적, 생성물 품질 검수, 동작 확인

코드를 직접 작성하지 않았다는 한계는 분명하다. 그래서 이 저장소가 보여주려는 것도 코드 자체가 아니다 — 무엇을 왜 만들었고, 어떤 위험을 먼저 막았고, 결과를 어떻게 검증했는지다.

## 핵심 설계 결정 4가지

상세는 **[01-build-journey.md](01-build-journey.md)** 참고.

1. **문제 풀을 자산으로** — 매 요청 LLM 생성 대신 사전 생성·검수 풀. 비용·지연·일관성 확보.
2. **RAG 인프라를 두 기능에 재사용** — 같은 embedding·pgvector가 문제 생성과 FAA Reference 사전을 동시에 만든다.
3. **Streamlit 상태 격리 원칙** — OAuth PKCE race condition을 `cache_resource`(전역 공유) vs `session_state`(세션별) 구분으로 해결.
4. **보안 모델 보존을 기준으로** — 출시 전 보안 하드닝과 Next.js 이주 결정 모두 "서버가 DB 접근을 중개한다"를 기준으로 내렸다.

## 한계

- 생성 문제의 정확성이 항상 보장되진 않는다 — 풀(pool) 문제는 검수를 거치지만, 라이브 생성·업로드 모드 문제는 사람 검수 없이 노출된다. 시험 준비 보조 도구이며 공식 자료를 대체하지 않는다.
- 품질은 수치 측정이 아니라 휴리스틱으로 관리 — 자동 패턴 스캔 + 샘플 인간 검수까지. 정식 평가셋(eval set)으로 정량화하지 않았다.
- 보안은 1인 audit, 외부 pentest 미실시.
